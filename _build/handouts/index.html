Collaborating with GitHub

Introduction

In this lab, you’ll be working with git and GitHub to collaborate on repos with a partner. One of the most powerful things about git/GitHub is the ability to work together like this. It takes some getting used to, but this lab will provide for multiple practices and can be repeated as many times as you need.

For this lab, you will need to work with a partner. The instructions are written for 2 people. If you have 3, things will be more complicated. You can have 1 author and 2 contributors or switch pairs throughout. Contributors should not do their steps at the same time. One contributor should work through them and then the other. In order for that to work, some author steps will need to be repeated as well, these steps are starred.

If you are having trouble finding a partner, reach out to a staff member.

Start by reading the “Overview of Collaborating” section for a refresher on the concepts. You’ll get into the hands-on portion in the “Practice” and “Conflicts” sections.

Which branch am I on?

In VS Code, you can see which branch you are on in the bottom left corner. You can also run git branch in your terminal to see a list of branches. The one you are currently working on will be starred and/or highlighted.

Overview of Collaborating

Some Important Terms

word

definition

repo

short for “repository”, which are the containers that git holds projects in

remote

all the code stored on GitHub is remote, away from your local machine, out in some GitHub database

local

the things you have on your computer, these files and folders aren’t available to your team unless you push them

main

the default base branch of your git repos (used to be called master)

branch

versions of your repository that are split off, like a copy, which you can edit without affecting “official” code

push

uploading local code to a remote repo

pull

downloading remote code to your local machine

pull request (PR)

a request made on GitHub to merge changes from one branch to another

checkout

the git command to switch to another branch (with the -b flag, it can also create new branches)

Summary

In a perfect world, everyone working on a repo would all be on the same page about what edits were happening where. In that case, the process would go something like this:

  • A project is held in a remote repository on GitHub, where everyone on the team has access to it

  • Everyone on the team has a local version of the project on their machine

  • When changes need to be made, everyone is aware of where those changes are, so people can avoid editing the same files on their local machines at the same time

  • A person making changes creates a local branch on their machine and makes the edits

  • That person pushes their local branch to the remote repo

  • On GitHub, that branch can be compared to the main branch and merged in if there are no conflicts (if there are conflicts, they can be resolved on GitHub)

  • Once the branch is merged in, everyone on the team should pull the main branch down to their local machine

  • If the changes affect anyone’s local branch, they can merge main into their local branch, otherwise next time they make a local branch, it will be based off main

When you have multiple people editing a repo, conflicts can arise. This often happens when 2, or more, people edit the same file without pulling each other’s changes. Usually one version will have been merged into main and when the next one attempts, there will be missing or mismatched code. There are other ways that conflicts can come about, but if you’re pulling changes regularly and communicating effectively, you shouldn’t run into them too often.

Forking

Today, we’ll be working directly in the same repos. This is often the practice for private repos. When contributing to an opensource project, or any other repo that you don’t have direct access to/authorship of, you should fork the repo to your GitHub account first. You can still branch and push to your repo, and then open a PR on the original repo. You can also pull the original repo’s code to your forked version.

Practice

Summary

In these scenarios, one partner should be author and one should be contributor. Keep your role throughout the exercise. Once you have completed the exercise, switch roles and repeat the steps. By the end, each partner should have had a chance to be author AND contributor for the entire exercise. Make sure you use different folder names when you repeat the steps so you don’t confuse them.

This exercise involves a lot of communication. Make sure you are doing every step together; timing will affect the outcomes. Share your screens back and forth as you move between steps so that everyone is on the same page.

Editing the Same File

  1. Figure out who will be author and who will be contributor

  2. author: in your terminal, navigate to where you’d like to save this project

  3. a: use mkdir to make a folder called “git-branch-practice” or similar, cd into that folder, and use touch to create an HTML file

  4. a: add an h1 tag containing your name to the file

  5. a: create a new repository on GitHub also called “git-branch-practice”

  6. a: in your terminal (make sure you’re in the “git-branch-practice” directory), run git init and then add and commit your code

  7. a: push your code to the new GitHub repo (you can copy the commands from GitHub)

  8. a: on GitHub, head to the Settings tab on your repo, click “Manage access” in the left menu, and then the green “Add people” button, add your partner(s) to your repository with “Read” permissions

    • You should now see your code on GitHub and have a URL to share with your partner, do so now. (Your partner will have also gotten an email invite from GitHub.)

  9. contributor: in your terminal, navigate to where you’d like to save this project

  10. c: use git clone <URL> to copy the contents (where <URL> is the repo’s URL without the brackets)

    • At this point, you both have the same code on your local machines, and it’s the same code that is in the remote repo

  11. c: in your terminal (in the “git-branch-practice” folder) run the following command to make a new branch called “feature/adding-h2”: git checkout -b feature/adding-h2

  12. c: add an h2 tag containing your name to the HTML file

  13. c: add and commit your changes

    • Now there is a local branch and changes to the HTML file that are only on the contributor’s machine. The next steps will help you push that code to the remote repo and merge it in to the main branch.

  14. c: use the following command to set up a remote branch, assign it as the place where your local branch should push to, and push your code to it: git push --set-upstream origin feature/adding-h2

  15. both: check the GitHub repo, you should both be able to see that a new branch has been added, explore the different views in GitHub to see what it can show you about the changes

  16. contributor: click the “branches” link on the repo’s GitHub page (left side of the same nav bar that the green “Code” button is in), and you should see a list of the branches, just main and feature/adding-h2

  17. c: on the right hand side of the page, there should be a gray “New pull request” button on the same row with your branch’s name, click on that button

    • The screen you’re seeing now allows you to name the PR and make any comments on it, this is great for tracking information about projects.

  18. c: create the pull request by clicking the green button

  19. author*: head to the “Pull request” tab of the repo and click into the PR your partner just made, there should be no conflicts

  20. a*: click the green “Merge pull request” button and then the “Confirm merge” button to merge the code into the main branch

    • The changes are now on the remote main branch, but they are not on anyone’s local main branch. (They are on the contributor’s local feature/adding-h2 branch.)

  21. both: in your terminal (make sure that you are in the right directory and that you are on the main branch, use git checkout main to switch back to it), run git status, note the message and figure out what it means

  22. both: run git pull origin main in your terminal

    • The remote changes are now on your local main branch!

If you have time at the end of this lab, you can repeat this process with as many changes as you would like. Try adding some more files and editing each other’s and your own code.

Adding Another File

Once a repository is established, it’s a much quicker process to push and pull changes.

  1. author: in your terminal (in the “git-branch-practice” folder, on the main branch), run the following command to make a new branch: git checkout -b feature/adding-css

  2. a: on your new branch, create a styles.css file and add a few styles (change a font/color/size/margin or 2)

  3. a: add and commit your code, then run git push --set-upstream origin feature/adding-css

    • Your CSS is now on a new remote branch in your repo!

  4. a: on GitHub, create a PR from your new branch and merge it into main

  5. both: in your terminal (in the “git-branch-practice” folder, on the main branch), run this command to get the CSS file: git pull origin main

Conflicts

Conflicts Ahead…

This section is purposefully a “how NOT to” section, so that you can practice resolving conflicts and recognize how they come about.

Resolving Conflicts Remotely

  1. contributor: in your terminal, run the following command to create a local branch: git checkout -b feature/adding-css

    Local vs Remote Branch Names

    The name of your local branch does not have to match the name of the remote branch, but it makes things a lot easier to keep track of.

  2. c: set the upstream origin (the place where your branch will push to) using this command: git push --set-upstream origin feature/adding-css

  3. c: make some changes to the CSS file

  4. c: add, commit, and push your changes

  5. c: head to the repo and create a PR

  6. author: approve and merge the PR

    Mistake Marker

    In a normal workflow, this is where you would pull the merged in changes to your local repo. But since we want to manufacture a conflict, we are not going to do that.

  7. a: from main create another branch using this command: git checkout -b conflict-branch

  8. a: make changes - different from your partner’s - to the CSS

  9. a: add, commit, and push (using git push --set-upstream origin conflict-branch) your code

  10. a: on GitHub, create a PR, you should get a warning that it will not merge automatically but that you can still create the PR

  11. a: on the middle of the following page, you should see a “Resolve conflicts” button, click it

    • In this view, you can see all of the files that have problems merging with each other. The lines with all the angle brackets denote which branch code is coming from. The lines of equals signs separate the code from each branch. In large files, it can become very cumbersome to fix merge conflicts, which is why communication is key.

  12. both: work through the file together to decide what changes to keep and then click the “Mark as resolved” (upper right) and then “Commit merge” buttons

  13. both: in your terminal (correct folder, main branch) run git pull to get the most recent remote version locally

    Continuing Work on Resolved Files

    New branches should be made from the most current version of main. You can also merge in branches locally if you need changes from main in your local branch.

You’ve resolved the issue!

When you have gone through the steps once, switch roles and go through them again!
